# Orpheus ‚Äî Design Context

This file is the confirmed design context for the impeccable design skills
(frontend-design, typeset, colorize, critique, ‚Ä¶). It is the source of truth for
_who Orpheus is for_ and _how it should feel_ ‚Äî things that cannot be inferred
from the code.

## Audience

One person ‚Äî the maintainer. Orpheus is a **personal, single-user tool**. There is
no onboarding, sharing, or multi-user surface, and audience-facing features are
deferred until "share-ready." Design for an expert who lives in the app daily.

## Use case

A macOS Electron IDE-style wrapper around the **Claude Code CLI**. The model is
Project ‚Üí Workspace, where each workspace owns one persistent libghostty terminal
running a `claude` session. The chrome around that terminal (sidebar, top bar,
settings, drawers, workspace lists) is what "the UI" refers to. The terminal's own
typography/colors come from the user's `~/.config/ghostty/config` and are out of
scope for app-chrome design work.

## Brand tone

**Refined dark terminal** ‚Äî a quiet-luxury developer tool. Readability comes first.
Character is expressed through restraint, not noise:

- Dark-primary, calm spacing, lighter text weight on dark surfaces.
- **Geist Pixel** is a deliberate _signature_, used only for the `Orpheus` wordmark
  and a small set of headers/eyebrow labels ‚Äî never for body or dense text.
- The interface must look **intentional under any of the selectable accent colors**,
  not just the default gold.

Avoid: loud retro/8-bit maximalism, personality-stripped "generic SaaS" flatness,
decorative fonts as body text, AI-slop tells (cyan-on-dark, purple‚Üíblue gradients,
glassmorphism, gradient text).

## Brand colors

- **Accent is user-selectable** (Settings ‚Üí Appearance): `gold` (default), `blue`,
  `teal`, `orange`, `pink`. It spans the full hue wheel (warm ‚Üí cool), so **neutrals
  must stay hue-neutral** ‚Äî the accent is the only chroma carrier. Do **not** hard-tint
  neutrals toward a single hue; if neutrals echo the accent at all, derive it from the
  _active_ accent via `color-mix`.
- **Themes:** `midnight` (default dark), `eclipse` (near-black), `daylight` (light).
- **GitHub PR-state palette is fixed** across themes (open `#1a7f37`, merged `#8250df`,
  closed `#cf222e`, draft `#6e7781`) to match github.com semantics.

## Design principles (project-specific)

- **Contrast floor: WCAG AA** ‚Äî text ‚â• 4.5:1 (‚â• 3:1 for large/bold), UI/icons ‚â• 3:1 ‚Äî
  against the _hardest_ surface it lands on, in **every theme √ó every accent**. Verify
  computationally, not by eye.
- **No alpha on text.** Opacity-dimmed text (`text-‚Ä¶/60`) is banned; use an explicit
  graded token instead.
- **One type scale.** No arbitrary `text-[Npx]`; everything maps to the modular `rem`
  scale with semantic tokens. Body never below ~15px.
- **Tokens, not raw colors.** Text/surfaces use theme tokens so they adapt per theme;
  no raw `text-zinc-*` etc. for theme-variant text.
