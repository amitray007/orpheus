### New engine — notable items (2026-07-03)

Re-review of staging PR #113's declarative migration engine (`src/main/db/*.ts`,
replacing `db.ts`). These are informational — surfaced for awareness, not open
findings requiring a fix:

1. **Whole-plan sync is not single-transaction-atomic.** `syncSchema()`'s plan can
   mix `rebuildTable` ops (which manage their own internal `BEGIN`/`COMMIT`) with
   individually-atomic DDL ops (`createTable`/`addColumn`/`dropColumn`/`addIndex`/
   `dropIndex`) run outside any outer transaction — a crash mid-plan can leave a
   partially-applied schema. Recoverable: a pre-rebuild backup is taken (unless
   in-memory or a genuinely fresh install) and re-running `syncSchema()` is
   idempotent, so the practical exposure is low. (`engine.ts:96-104`)
2. **Carried-over columns lose CHECK/PK text on rebuild.** When a table rebuild
   preserves a live column not declared in the desired schema (a "carry-over"),
   the shadow-table `ADD COLUMN` fragment is reconstructed from introspection,
   which only exposes type/notNull/default — not the column's original CHECK or
   PRIMARY KEY clause. Data is preserved; any CHECK/PK on that specific column is
   not. (`rebuild.ts:22-28`)
3. **`normalizeDefault`'s single-paren-strip could false-positive a rebuild
   (latent).** It strips exactly one layer of enclosing parens to compare live
   vs. desired DEFAULT expressions cosmetically. A default that legitimately
   needs two layers of parens would compare unequal to itself post-round-trip
   and could trigger an unnecessary rebuild. No such default exists in the
   current schema — flagged as latent risk, not an active bug. (`diff.ts:54-64`)
4. **`verify-migration-engine.ts` validates logic, not the production binding.**
   The 91-assertion harness runs the engine under Node's built-in `node:sqlite`
   (`DatabaseSync`), not the `better-sqlite3` binding the packaged app actually
   uses. The engine's SQL/logic is exercised end-to-end, but ABI-specific
   behavior differences between the two bindings (if any) are not covered — a
   real coverage gap worth closing (e.g. an Electron-ABI smoke pass) before
   leaning on this harness as the sole migration safety net.
5. **No SQL-injection surface.** No `SELECT *`-style dynamic composition and no
   un-parameterized user data reaches SQL — all DDL identifiers (table/column/
   index names) come from internal schema constants (`schema.ts`), never from
   external input. Injection-safe by construction.
